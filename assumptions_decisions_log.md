# Assumptions and Decisions Log

This document tracks assumptions, decisions, and open questions made throughout the project. The goal is to make the workflow transparent and reproducible.

| Date | Area | Assumption or Decision | Reason | Risk | Open Questions |
|---|---|---|---|---|---|
| 05/24/2026 | Project setup | Raw data will never be overwritten. Cleaned data will be saved separately. | Preserves reproducibility and allows comparison between raw and cleaned data. | Requires more storage and careful file organization. | None. |
| 05/24/2026 | Project scope | This project will use mock or public data only. | Avoids confidentiality issues and allows the project to be shared. | Mock data may not capture all real-world complexity. | What real issues would appear in the organization's actual data? |
| 05/24/2026 | Business framing | the project will focus on fleet maintenance, fuel, vehicles, and vendors. | These areas are relevant to fleet management and business excellence. | Scope could become too large if not controlled. | Which area would create the most business value first? |
| 05/24/2026 | Data quality | Questionable records will be flagged as exceptions instead of automatically corrected whenever the correct value is uncertain. | Prevents risky assumptions and allows business review. | More records may require manual review, at least initially. | Who should own exception resolution in a real organization? |
| 05/24/2026 | Data storage | The public datasets will be stored separately from the mock raw files. | Ensures that the project reproducible and different approaches are testable. | More storage space is needed for both the original and the mock datasets. | What measures are in place in organizations to prevent changing original data? |
| 05/24/2026 | Dataset generation | A hybrid approach will be taken in generating the mock datasets. | Using public datasets as inspiration but creating mock datasets gives us the ability to control specific problems to solve, as well as base our mock data on real-world datasets. | While some information may be based on real-world datasets, the scenarios created and some part of the datasets will be fabricated, which might lack real-world representation. | What would be the best approach in simulating potential real-world data errors? |
| 05/24/2026 | Mock dataset limitations | The compatibility issues in mock datasets will be ignored (mentioned in docs/mock_dataset_design.md). | The mock datasets can support the scope of this project, but does not aim to represent real-life fleet specifications. | There will be unavoidable and obvious compatibility issues, but they should not interfere with the goals of the project. | None. |
| 05/24/2026 | SQL ingestion | The raw CSV files will be uploaded to the SQL server without cleaning them up. | The goal is to preserve the original messy data before making transformations. | This reduces the risk of losing the messy, original data. It should not introduce risks. | None. |
| 05/27/2026 | Clean layer design | Cleaned data will first be created as SQL views instead of physical tables. | Views keep the cleaning logic transparent, easy to revise, and reproducible before creating permanent clean tables. | Views may be slower than physical tables for larger datasets and may need to be materialized later. | At what point should clean views be converted into physical clean tables? |
| 05/27/2026 | Vehicle key logic | `asset_no` will be used as the primary clean vehicle key because it is complete and unique in the mock vehicle master. | Profiling showed that `asset_no` is more reliable than `vehicle_id`, which has missing values. | In a real organization, `asset_no` may not always be the true system-of-record identifier. | Which vehicle identifier would be considered authoritative in a real fleet system? |
| 05/27/2026 | Missing vehicle IDs | Missing `vehicle_id` values may be temporarily derived from `asset_no`, but those records will still be flagged for review. | This allows records to remain usable in the clean layer while making the missing source identifier visible. | Derived IDs may create a false sense of confidence if users mistake them for original source-system IDs. | Should derived vehicle IDs be allowed in production reporting? |
| 05/27/2026 | Vendor key logic | `vendor_id` will be used when available; missing vendor IDs may receive a controlled derived key but should remain flagged. | This allows vendor records with missing IDs to remain available for review and limited analysis without overwriting the raw data. | Derived vendor keys may not match future source-system corrections unless a crosswalk is maintained. | Who should approve derived or surrogate vendor keys in a real workflow? |
| 05/27/2026 | Categorical standardization | Fields like fuel type, vehicle status, transmission type, partner type, and boolean flags will be standardized using SQL `CASE` logic. | Standardization makes reporting more consistent and prevents equivalent values from being split across multiple categories. | Hard-coded mappings may miss new or unexpected source values. | Should standardized value mappings eventually be stored in lookup tables instead of SQL `CASE` statements? |
| 05/27/2026 | Exception handling | Records can still appear in clean views even when flagged, but their `clean_record_status` will indicate whether they are OK, usable with warning, or review required. | This avoids deleting potentially useful records while still warning users about data quality concerns. | Users may ignore warning flags if dashboards do not make them visible. | Which metrics should exclude records marked `REVIEW_REQUIRED`? |
| 05/27/2026 | Duplicate identifiers | Duplicate VINs, registration numbers, and vendor IDs will not be silently resolved and instead be flagged for review. | Duplicate identifiers can indicate entity-resolution problems that require business context. | Leaving duplicates unresolved may limit confidence in master-data reporting. | Who should decide which duplicate record is authoritative? |
| 05/27/2026 | Vendor normalization | Normalized vendor names are useful for identifying likely duplicates, but they are not enough by themselves to automatically merge vendors. | Similar vendor names may represent the same vendor, but automatic merging could incorrectly combine separate entities. | Manual review is slower and may require business input. | Should the project eventually include a vendor alias or crosswalk table? |
